# Sub-project B / T5 — Why we ship NO patch to `lib/tagcanvas.js`

**Date:** 2026-05-03  | **Author:** PM (claude-opus-4.7-xhigh)
**Audit context:** GPT-5.5 R3+R4 cross-audits of B's plan

## TL;DR

Upstream issue [MikeCoder#39][i39] reports tag names with the `+`
character displaying as `&#43;` on the canvas. The B spec initially
proposed patching `lib/tagcanvas.js` to decode HTML entities in the
text-acquisition path. The cross-auditor (GPT-5.5) caught a real
double-decode regression risk in that patch (`A&copy;B` → `A©B`),
and a live reproduction in our fixture proved the bug **doesn't trigger
with stock hexo + TagCanvas v2.9 in modern Chrome**. We therefore ship
NO patch — only regression tests + a README troubleshooting note.

[i39]: https://github.com/MikeCoder/hexo-tag-cloud/issues/39

## Live reproduction (2026-05-03)

Setup:
- Repo at `feature/v3-overhaul` after sub-project A
- Fixture site `tests/fixtures/hexo-site/` extended with tags
  `C++`, `A&B`, `quote"tag` plus the existing CJK / Cyrillic set
- Headless Chromium via Playwright

Result — TagCanvas's per-tag `text_original` field after `Start()`:

| Config tag    | TagCanvas `text_original` (observed) |
|---------------|--------------------------------------|
| `C++`         | `"C++"`                              |
| `A&B`         | `"A&B"`                              |
| `quote"tag`   | `"quote\"tag"`                       |
| `中文`        | `"中文"`                             |
| `日本語`      | `"日本語"`                           |
| `한국어`      | `"한국어"`                           |
| `Привет`      | `"Привет"`                           |

All literal — no HTML entities, no mangling. The hexo `tagcloud()`
helper outputs `<a>...&#43;</a>`-style HTML; the browser's HTML parser
decodes that into a DOM text node whose `innerText` returns the literal
character. TagCanvas v2.9 reads that path correctly.

## Why issue #39 still fires for some users

The reporter's `npm list` includes `cheerio@1.0.0-rc.12` and
`hexo-asset-image@1.0.0`. Those plugins post-process generated HTML
through cheerio's parser, which can re-serialise text with entities
re-introduced — depending on plugin order, the `<a>` text may end up
as the LITERAL 5-character string `&#43;` in the DOM (not an entity
that the browser decodes). At that point TagCanvas reads back `&#43;`
because it IS the text content, not an encoding of `+`.

Net: the bug is in the user's cheerio-using plugin chain, not in
TagCanvas or hexo-tag-cloud. Patching TagCanvas to "decode entities
just in case" is the wrong fix and introduces the
`A&copy;B` → `A©B` regression the cross-auditor caught.

## What we ship instead

1. **`tests/e2e/smoke.spec.js`** — extended with exact-match assertions
   that `taglist.text_original` contains `"C++"`, `"A&B"`, `"quote\"tag"`
   and the CJK / Cyrillic tags, plus a non-zero opaque pixel-count
   assertion. These tests would flag any regression in TagCanvas's
   text acquisition if a future maintainer accidentally introduces one.

2. **README "Troubleshooting"** — one-paragraph note pointing users at
   their cheerio-based hexo plugin chain when they see `&#NN;` codes
   on the canvas.

3. **`lib/tagcanvas.js`** — byte-identical to upstream TagCanvas v2.9.
   Easy to compare with upstream when newer TagCanvas releases land.

## If a real reproduction surfaces in the future

The B plan's "out of scope" guard documents the path: file a
follow-up note, narrowly target the failing case, and cover the
double-decode risk (`A&copy;B` must survive untouched) with a guard
test before merging the patch. Do NOT chase a TagCanvas refactor —
that's an explicit non-goal of v3.0.0.
